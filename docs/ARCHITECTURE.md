# Napkin Architecture & Development Guide

This document provides a high-level overview of the Napkin CLI architecture for human developers and AI agents.

## Core Design Philosophy

Napkin is designed to be a **lean, zero-dependency generator**. 
1. **Developer Experience (DX)**: Scaffolding should be instantaneous.
2. **Runtime Simplicity**: Generated services must use only native Node.js libraries to ensure they run anywhere with minimal overhead.
3. **Transparency**: The generated code is explicitly provided in the CLI as template strings to avoid complex file system lookups and to make the "runnable sketch" logic easy to inspect.

## Project Structure

- `src/cli.ts`: The entry point. It uses `commander` to manage command registration.
- `src/commands/`: Contains the implementation of top-level commands.
  - `init.ts`: Scaffolds the `napkin.yaml` manifest.
  - `generate/`: Sub-group for generation commands.
    - `service.ts`: The primary logic for scaffolding REST services.
- `dist/`: The build output. Compiled via `tsup` into ESM.

## Manifest (`napkin.yaml`)

The `napkin.yaml` file is the source of truth for the local architecture sketch. 
When a new service is generated:
1. The physical files are created in `services/<name>/`.
2. The manifest is updated with the service's ID, path, expected ports, and data fields.

## Generated Service Architecture

Services generated by Napkin follow a strict "No Dependency" rule:
- **Server (`index.js`)**: Uses native `node:http`. It handles basic routing for REST operations (GET, POST, PUT, DELETE).
- **Storage (`storage.js`)**: Uses native `node:fs/promises` for CSV persistence.
- **Persistence**: Data is saved in a local `data.csv` file using absolute paths (via `__dirname`) to ensure the service is portable across different execution contexts.

## Instructions for AI Agents

When modifying this codebase:
1. **Maintain Zero-Deps**: Do not add dependencies to the *generated* services. Use only Node.js built-ins.
2. **Atomic Updaters**: When modifying `napkin.yaml`, always load the full file, update the object, and write it back using `js-yaml` to maintain consistency.
3. **Template Strings**: The service templates are currently inlined in `src/commands/generate/service.ts`. If they grow significantly, consider moving them to a dedicated `src/templates` directory and ensuring `tsup` (or a pre-build step) includes them in the distribution.
